Software Design

Design Patterns Used

🎯 Strategy Pattern

SplitStrategy interface with 3 implementations: EqualSplitStrategy, ExactSplitStrategy, PercentageSplitStrategy. Selected at runtime based on expense type.

🏭 Factory Pattern

SplitStrategyFactory.getStrategy(SplitType) creates the correct strategy instance via switch-case. Client code is decoupled from concrete classes.

📐 Template Method

BaseServiceImpl<T, ID> provides generic CRUD operations inherited by all 5 service implementations.

📦 DTO Pattern

13 DTO classes decouple API shapes from internal entities. Request DTOs accept input, Response DTOs shape output.

🔄 Mapper Pattern

5 static Mapper classes convert Entity ↔ DTO without polluting model classes with serialization logic.

💉 Dependency Injection

Guice Module.java binds 5 service interfaces to their implementations. Controllers receive dependencies via @Inject.

Core Algorithm

Split Strategies — Deep Dive

⚖️

EQUAL Split

amount / participants.size()

₹900 ÷ 3 people = ₹300 each

Payer's split set to ₹0 (they don't owe themselves)

📝

EXACT Split

User specifies exact values[]

₹900 → [₹200, ₹400, ₹300]

Validation: sum of values must equal total amount

📊

PERCENTAGE Split

amount × percentage / 100

₹1000 → [50%, 30%, 20%]
= [₹500, ₹300, ₹200]

Validation: percentages must sum to 100

Algorithms

Balance Tracking & Settlement

Balance Recalculation (addTransaction)

Smart net-off logic when adding a debt:

addTransaction(debtor, creditor, amount):

  // Case 1: Reverse debt exists
  if FIND (creditor→debtor):
    if reverse > amount:
      reverse -= amount        ← reduce
    elif reverse == amount:
      DELETE reverse            ← cancel out!
    else:
      DELETE reverse
      CREATE (debtor→creditor,  ← flip direction
              amount - reverse)

  // Case 2: Same direction
  elif FIND (debtor→creditor):
    existing += amount          ← accumulate

  // Case 3: New debt
  else:
    CREATE new balance

Settlement Simplification (Greedy)

Minimizes number of transactions:

1. Compute net balance per user
   (sum credits - sum debts)

2. Partition into two max-heaps:
   • Debtors  (negative balances)
   • Creditors (positive balances)

3. While both heaps non-empty:
   • Pop largest debtor & creditor
   • Settle min(debt, credit)
   • Reinsert remainder

4. Return settlement list

⏱ Complexity

O(N log N) where N = users with non-zero balance. Greedy approach ensures minimum transactions.

Worked Example: Trip Expenses

Scenario

3 friends — Alice, Bob, Charlie — on a trip.

Expense Paid By Amount Split
Hotel Alice ₹3000 EQUAL
Dinner Bob ₹900 EQUAL
Taxi Charlie ₹600 EQUAL

Split Calculations

Hotel: ₹3000 ÷ 3 = ₹1000 each
→ Bob owes Alice ₹1000, Charlie owes Alice ₹1000

Dinner: ₹900 ÷ 3 = ₹300 each
→ Alice owes Bob ₹300, Charlie owes Bob ₹300

Taxi: ₹600 ÷ 3 = ₹200 each
→ Alice owes Charlie ₹200, Bob owes Charlie ₹200

Net Balances (after net-off)

User Net Status
Alice +₹1000 - ₹300 - ₹200 = +₹500 Owed ₹500
Bob -₹1000 + ₹300 - ₹200 = -₹900 Owes ₹900
Charlie -₹1000 - ₹300 + ₹200 = -₹1100 ... (net differs) ...

Summary

Key Takeaways

🏗️

Clean Architecture

5-layer separation: Presentation → Transfer → Business Logic → Data Access → Persistence. Each layer has a single responsibility.

🧩

6 Design Patterns

Strategy, Factory, Template Method, DTO, Mapper, and Dependency Injection — used where they add real value, not for the sake of patterns.

🧮

Smart Algorithms

Net-off balance tracking avoids redundant records. Greedy max-heap settlement minimizes the number of money transfers needed.

🔗

Type-Safe Full Stack

Java DTOs mirror TypeScript interfaces. 12 RESTful endpoints with consistent JSON error responses via global ErrorHandler.
